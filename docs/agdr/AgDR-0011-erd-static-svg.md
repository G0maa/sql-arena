# AgDR-0011 — Schema ERD rendered as a pre-generated static SVG

**Status**: Accepted
**Date**: 2026-06-13
**Author**: Hisham (Tech Lead)

> In the context of the contestant page (#11), facing the need to show contestants the full
> `seed` schema — tables, every column with its type, and the foreign-key relationships — so
> they know what to query and what to index, I decided to render the ERD as a **pre-generated
> static SVG** produced by a dependency-free Node generator (`scripts/gen-erd.mjs`) and served
> as a static asset, to achieve a genuine detailed diagram while keeping the page a pure
> static file with no runtime library, accepting that the SVG must be regenerated by hand when
> the schema changes.

## Context

The page is locked to **plain HTML/JS/CSS — no framework, no build step, works offline**
(continues AgDR-0003's single-worker simplicity ethos and the AgDR-0004 sandbox model, where
contestants add indexes via `setup_sql`). A contestant can't write a query — let alone decide
*what to index* — without knowing the table names, column names, column types, and the FK join
paths. The seed schema has only PK indexes; the FK columns (`product.category_id`,
`orders.customer_id`, `order_details.product_id`, `order_details.order_id`) are unindexed and
are precisely the optimization surface. So the ERD must show **all of that, clearly, on page
open**.

A "detailed ERD" means entity boxes with full attribute lists **and** relationship lines —
which is exactly what a diagramming tool like Mermaid's `erDiagram` produces. The tension is
the no-framework/offline constraint.

## Options Considered

| Option | Pros | Cons |
|--------|------|------|
| **Runtime Mermaid (vendored `mermaid.min.js`)** | Trivial authoring; robust layout | ~2.8 MB JS dependency shipped to every contestant; reopens the locked "no framework" decision; runtime render cost |
| **Pre-rendered Mermaid → SVG inline** | Detailed diagram, no runtime dep | Needs `mmdc` + headless Chromium to generate — **not available** in this environment |
| **Hand-authored static SVG** | No dependency; full control | Laying out 5 boxes + ~22 rows + routed connectors by hand is fragile and error-prone to maintain |
| **Dependency-free SVG generator script** | No runtime dep; pure static/offline page; regenerable from a schema model; the math (box sizing, row anchors, connector routing) is done in code, not by hand | Regeneration is a manual authoring step (`node scripts/gen-erd.mjs`) when the schema changes; layout coordinates are hand-tuned |
| **Pure text reference panel (no diagram)** | Simplest | Not a diagram — the operator explicitly asked for a detailed ERD showing relationships |

## Decision

Chosen: **dependency-free SVG generator script**, because it delivers a real detailed ERD
(tables, all columns + types, PK/FK badges, FK→PK relationship arrows) while the page stays a
pure static file with **zero runtime library** — honoring the offline/no-framework constraint
for everything the contestant's browser loads. The Mermaid-style "diagramming tool" effort
moves to authoring time and is expressed as committed, reviewable code rather than a vendored
blob.

- Generator: `scripts/gen-erd.mjs` — a self-contained Node ESM script (no npm deps) holding the
  schema model and emitting `public/erd.svg`.
- Artifact: `public/erd.svg` — committed, served by `ServeStaticModule`, embedded in the page
  via `<img src="erd.svg">`.
- The schema model in the generator is the single source for the diagram; it must be updated
  (and the script re-run) if `db/sql/seed.sql` changes.

## Consequences

- The page gains a full-width "Database schema" panel, visible on open, with the ERD.
- No new runtime dependency; AgDR-0003/0004's no-framework/offline stance is preserved for
  served code. (The generator is dev-time tooling, not shipped to the browser.)
- **Drift risk**: the SVG is not auto-generated at build/serve time, so a schema change that
  isn't followed by re-running the generator will leave the diagram stale. Mitigation: the
  generator is small and the schema is stable; a future improvement could assert the model
  against `information_schema` in a test.
- If a richer/interactive diagram is ever needed, revisit with the vendored-Mermaid option.
